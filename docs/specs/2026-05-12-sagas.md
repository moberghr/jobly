# Spec: Sagas — long-lived correlated state for message-driven workflows

> **Implementation note (2026-05-13):** the design pivoted during build. This spec is preserved as the original proposal; the shipped surface is documented in `CLAUDE.md` §8.9 and `website/docs/features/sagas.md`. The major deltas:
>
> - **Delete-on-completion**, not keep-and-expire. No `IsComplete` / `CompletedAt` / `ExpireAt` columns on `SagaState`; the proxy removes the row in the same `SaveChanges` when `MarkCompleted()` is called. Unique index is plain `(Type, CorrelationKey)`, not a filtered partial index. `WarpConfiguration.SagaExpirationTimeout` is not added.
> - **Activity log** via a `SagaJobLink (SagaId, JobId, CreatedAt)` extension table — added beyond the original spec.
> - **Dashboard UI** (`/warp/sagas`, `/warp/sagas/{id}`) + `ISagaQueryService` / `ISagaCommandService` + REST endpoints — originally deferred to v2, shipped in v1.
> - **Typed correlation keys** via `Saga<TKey>` for `string` / `Guid` / `int` / `long` — originally deferred, shipped in v1. Canonicalization via `SagaCorrelationKeyConverter`; storage stays a single `varchar(200)`.
> - **`ITimeoutMessage`** + auto-scheduling on `Publisher.Publish` — originally deferred to v2/v3, shipped in v1. Timeout-after-completion silently sets `Outcome = Deleted`.
> - **`SagaPiiCheck`** startup-time guard against PII-named `[Correlate]` properties — not in original spec.
> - **`AddSagaHandler<T>()`** has an idempotency guard against double-registration — throws rather than silently double-processing.

## Problem

Warp's existing primitives (`IJob`, `IMessage`, `Batch`, `ParentJobId` continuations) all assume the workflow is known up front: jobs run, possibly trigger continuations, and finish. None of them handle the canonical *distributed workflow* shape:

> A business process spans hours-to-days. External messages arrive over time, **in any order**, all referencing the same business identifier (`OrderId = "O-12345"`). The next step depends on which messages have already arrived. We need persisted, **correlated** state that survives between message arrivals, and **only one handler in flight per correlation** at a time.

This is the territory where MassTransit, NServiceBus, and Wolverine differentiate themselves from pure DB-backed schedulers (Hangfire, Quartz). Warp can compete here because it already has the storage layer, optimistic-concurrency interceptor, and a distributed-lock primitive — what's missing is the saga abstraction stitched on top.

Reference designs studied:

- **Wolverine** — `Saga` base class, attribute-driven correlation, single state row per instance with optimistic concurrency, mutex over the correlation key for serialization. Closest to Warp's existing primitives.
- **MassTransit** — state-machine DSL (`MassTransitStateMachine<TInstance>`) over a saga repository. More expressive, much more API surface.
- **NServiceBus** — `Saga<TData>` with `ConfigureHowToFindSaga`. Storage-first, similar shape to Wolverine but more verbose.

Warp's v1 takes Wolverine's "code-first" shape (no state-machine DSL) because it slots cleanly into the existing `IMessageHandler<T>` dispatch path and the existing `[Mutex]` primitive — see *Solution* below.

## Solution (v1)

### Separate `Saga` entity, opt-in addon

A new addon `opt.AddSagas()` mirroring `AddConcurrency()` / `AddCircuitBreaker()` (`src/core/Warp.Core/Concurrency/ConcurrencyServiceConfiguration.cs:9`). Contributes a single `Saga` entity via `EntityConfigurators` so non-users pay nothing in schema or runtime. Lives in `src/core/Warp.Core/Sagas/` next to the other addons. Provider-agnostic — no Npgsql / SqlClient references (§5.1).

### Public API

```csharp
public abstract class Saga
{
    public Guid Id { get; set; } = Guid.NewGuid();

    public string CorrelationKey { get; set; } = string.Empty;

    public bool IsComplete { get; set; }
}

[AttributeUsage(AttributeTargets.Property, Inherited = false)]
public sealed class CorrelateAttribute : Attribute;

[AttributeUsage(AttributeTargets.Class, Inherited = false)]
public sealed class StartsSagaAttribute : Attribute;

public interface ISagaHandler<TSaga, in TMessage>
    where TSaga : Saga, new()
    where TMessage : class, IMessage
{
    Task HandleAsync(TSaga saga, TMessage message, CancellationToken cancellationToken);

    // Invoked when a non-[StartsSaga] message arrives for an unknown correlation key.
    // Default implementation sets IJobContext.Outcome = Failed with a clear log.
    // Override to silently ignore, log-and-skip, or perform compensation.
    Task NotFoundAsync(TMessage message, IJobContext context, CancellationToken cancellationToken) =>
        Task.CompletedTask;
}
```

User code:

```csharp
public class OrderSaga : Saga
{
    public string OrderId { get; set; } = string.Empty;
    public bool PaymentCaptured { get; set; }
    public bool InventoryReserved { get; set; }
}

[StartsSaga]
public class OrderPlaced : IMessage
{
    [Correlate]
    public string OrderId { get; set; } = string.Empty;
}

public class PaymentCaptured : IMessage
{
    [Correlate]
    public string OrderId { get; set; } = string.Empty;
}

public class OrderWorkflow :
    ISagaHandler<OrderSaga, OrderPlaced>,
    ISagaHandler<OrderSaga, PaymentCaptured>
{
    public Task HandleAsync(OrderSaga saga, OrderPlaced msg, CancellationToken ct)
    {
        saga.OrderId = msg.OrderId;

        return Task.CompletedTask;
    }

    public Task HandleAsync(OrderSaga saga, PaymentCaptured msg, CancellationToken ct)
    {
        saga.PaymentCaptured = true;
        if (saga is { PaymentCaptured: true, InventoryReserved: true })
        {
            saga.IsComplete = true;
        }

        return Task.CompletedTask;
    }
}

// Registration
services.AddWarpWorker<AppDbContext>(opt =>
{
    opt.UsePostgreSql();
    opt.AddSagas();                                // contributes Saga entity, registers proxies, store
});
services.AddSagaHandler<OrderWorkflow>();          // discovers ISagaHandler<,> interfaces it implements
```

### Storage entity

```csharp
namespace Warp.Core.Data.Entities;

public class Saga : IConcurrencyToken
{
    public Guid Id { get; set; }

    public string Type { get; set; } = string.Empty;          // AssemblyQualifiedName of TSaga

    public string CorrelationKey { get; set; } = string.Empty;

    public string StateJson { get; set; } = string.Empty;     // serialized TSaga

    public bool IsComplete { get; set; }

    public DateTime CreatedAt { get; set; }

    public DateTime UpdatedAt { get; set; }

    public DateTime? CompletedAt { get; set; }

    public DateTime? ExpireAt { get; set; }                   // set on completion; cleanup uses this

    public Guid Version { get; set; }                         // IConcurrencyToken (interceptor bumps automatically)
}
```

Entity configuration in `ServiceConfiguration.AddSagaStateEntity(ModelBuilder, string?)` (registered only when `AddSagas()` is called, mirroring `AddConcurrencyLimitEntity`):

- PK on `Id`
- **Unique index** on `(Type, CorrelationKey)` *where* `IsComplete = false` — guarantees one live instance per correlation. Filtered index on PG (`HasFilter("\"is_complete\" = false\"")`) and SQL Server (`HasFilter("[IsComplete] = 0")`).
- Index on `(IsComplete, ExpireAt)` for cleanup
- `StateJson` mapped as `text`/`nvarchar(MAX)`
- Default schema = `WarpConfiguration.Schema` via `entity.Metadata.SetSchema(schema)` (§2.5)

### Dispatch flow

A `SagaHandlerProxy<TSaga, TMessage> : IMessageHandler<TMessage>` is registered as the `IMessageHandler<TMessage>` whenever the user calls `AddSagaHandler<THandler>()`. The proxy is what `MessageRouter` discovers and what `JobDispatcher` invokes. The user's `ISagaHandler<TSaga, TMessage>` instance is what the proxy calls into.

Per-message flow inside the proxy's `HandleAsync(message, ct)`:

```
1. correlationKey = ExtractCorrelationKey(message)            // [Correlate] property on TMessage
   - cached per-type reflection lookup (ConcurrentDictionary<Type, PropertyInfo>)
   - if no [Correlate] property → throw SagaConfigurationException at startup, not at dispatch
2. lockKey = $"warp:saga:{typeof(TSaga).FullName}:{correlationKey}"
3. handle = await _semaphoreProvider.TryAcquireAsync(lockKey, maxCount: 1, TimeSpan.Zero, ct)
4. handle == null:
   - _jobContext.Outcome = { State = Enqueued, ScheduleTime = now,
                              ClearHandlerType = true,
                              LogMessage = $"Requeued — saga '{TSaga.Name}' busy for '{correlationKey}'" }
   - return
5. try:
     a. saga = await _sagaStore.Load<TSaga>(correlationKey, ct)
     b. saga == null:
        - if !TMessage has [StartsSaga]:
            - jobContext.Outcome = { State = Failed, LogMessage = $"No saga for '{TSaga.Name}' with key '{correlationKey}'" }
              (set as a default outcome BEFORE calling NotFoundAsync, so a no-op override yields Failed)
            - await _innerHandler.NotFoundAsync(message, jobContext, ct)
              (override may overwrite jobContext.Outcome — e.g. set to Deleted for silent ignore, or null to swallow)
            - release mutex and return; do not invoke HandleAsync, do not save saga
        - else: saga = new TSaga { CorrelationKey = correlationKey }
        - mark as Added in change tracker
     c. await _innerHandler.HandleAsync(saga, message, ct)
     d. await _sagaStore.Save(saga, ct)
        - if Added: insert with CreatedAt = now, UpdatedAt = now
        - if Modified: update StateJson, UpdatedAt = now (interceptor bumps Version)
        - if saga.IsComplete: set CompletedAt = now, ExpireAt = now + SagaExpirationTimeout
   finally:
     await handle.DisposeAsync()
6. catch DbUpdateConcurrencyException:
   - defense-in-depth: with the mutex held this should not fire, but if it does (e.g. semaphore provider released early)
     set outcome = { Enqueued, ScheduleTime = now, ClearHandlerType = true,
                      LogMessage = $"Saga '{key}' version conflict — requeued" }
```

The mutex is the primary serialization mechanism — picked deliberately in brainstorming. Optimistic concurrency on the `Version` column is defense-in-depth in case the mutex is somehow not held (e.g. provider failure, future refactor). Both layers are cheap; the row-version is bumped automatically by the existing `SaveChangesConcurrencyTokenInterceptor` (`src/core/Warp.Core/Data/Interceptors/SaveChangesConcurrencyTokenInterceptor.cs:8`).

### `ISagaStore`

```csharp
public interface ISagaStore
{
    Task<TSaga?> Load<TSaga>(string correlationKey, CancellationToken ct)
        where TSaga : Saga, new();

    void Add<TSaga>(TSaga saga)
        where TSaga : Saga;

    void Update<TSaga>(TSaga saga)
        where TSaga : Saga;

    Task SaveChangesAsync(CancellationToken ct);
}
```

Concrete `SagaStore<TContext> : ISagaStore`, scoped DI lifetime:

- `Load` — `await context.Set<Saga>().Where(x => x.Type == typeName && x.CorrelationKey == key && !x.IsComplete).FirstOrDefaultAsync(ct)`, then JSON-deserialize `StateJson` into `TSaga`. Returns `null` if not found. **No row lock**: the mutex (acquired upstream in `SagaHandlerProxy`) provides cross-process serialization on `(SagaType, CorrelationKey)` — plain LINQ is sufficient. Defense-in-depth is the optimistic `Version` token bumped automatically by `SaveChangesConcurrencyTokenInterceptor`.
- `Add` — JSON-serializes `TSaga` to `StateJson`, sets `CreatedAt`, `UpdatedAt`, `Id` from the in-memory saga's `Id`, adds to context.
- `Update` — JSON-serializes `TSaga` to `StateJson`, sets `UpdatedAt`, marks the tracked entity as `Modified`. The interceptor handles `Version`.
- `SaveChangesAsync` — delegates to context. Caller controls the save boundary (§5.7). The proxy calls it after the handler returns.

JSON serialization uses a single `JsonSerializerOptions` cached on `SagaStore<TContext>` — same shape as `Job.Message` handling in `Publisher.cs:105`.

### Cleanup

Completed sagas (`IsComplete = true`, `ExpireAt <= now`) are deleted by `ExpirationCleanup` in the next pass. This requires extending the existing `ExpirationCleanup` server task to also clean Saga rows — already iterates over `Job`/`JobLog`/`RecurringJobLog`, so the change is additive: one more `ExecuteDeleteAsync` against `_context.Set<Saga>()`.

`SagaExpirationTimeout` is a new field on `WarpConfiguration` (default `TimeSpan.FromDays(1)`, mirroring `JobExpirationTimeout`).

When `AddSagas()` is not registered, the `Saga` entity is not in the model and the cleanup branch is no-op — guarded by `context.Model.FindEntityType(typeof(Saga))` check, returning early. Avoids a hard dependency on the addon from a Worker-side server task.

### `AddSagas()` and `AddSagaHandler<THandler>()`

```csharp
public static class SagaServiceConfiguration
{
    public static IWarpBuilder<TContext> AddSagas<TContext>(this IWarpBuilder<TContext> builder)
        where TContext : DbContext
    {
        builder.Configuration.EntityConfigurators.Add(ServiceConfiguration.AddSagaStateEntity);

        builder.Services.AddScoped<ISagaStore, SagaStore<TContext>>();
        builder.Services.AddSingleton<SagaCorrelationCache>();
        // SagaHandlerProxy<TSaga, TMessage> registrations happen per-handler in AddSagaHandler

        return builder;
    }

    public static IServiceCollection AddSagaHandler<THandler>(this IServiceCollection services)
        where THandler : class
    {
        var handlerType = typeof(THandler);
        services.AddScoped<THandler>();

        var sagaInterfaces = handlerType.GetInterfaces()
            .Where(i => i.IsGenericType && i.GetGenericTypeDefinition() == typeof(ISagaHandler<,>))
            .ToList();

        if (sagaInterfaces.Count == 0)
        {
            throw new InvalidOperationException(
                $"{handlerType.Name} does not implement ISagaHandler<,>.");
        }

        foreach (var iface in sagaInterfaces)
        {
            var sagaType = iface.GenericTypeArguments[0];
            var messageType = iface.GenericTypeArguments[1];

            // Register the user-typed interface so the proxy can resolve it
            services.AddScoped(iface, sp => sp.GetRequiredService(handlerType));

            // Register the proxy as IMessageHandler<TMessage>
            var proxyType = typeof(SagaHandlerProxy<,>).MakeGenericType(sagaType, messageType);
            var messageHandlerType = typeof(IMessageHandler<>).MakeGenericType(messageType);
            services.AddScoped(messageHandlerType, proxyType);
        }

        return services;
    }
}
```

`SagaCorrelationCache` is a singleton `ConcurrentDictionary<Type, PropertyInfo>` that resolves the `[Correlate]` property for a message type once. Throws on registration if a message reaching the proxy has no `[Correlate]` property.

### `SagaHandlerProxy<TSaga, TMessage>`

```csharp
public class SagaHandlerProxy<TSaga, TMessage> : IMessageHandler<TMessage>
    where TSaga : Saga, new()
    where TMessage : class, IMessage
{
    private readonly ISagaHandler<TSaga, TMessage> _inner;
    private readonly ISagaStore _store;
    private readonly IWarpSemaphoreProvider _semaphoreProvider;
    private readonly IJobContext _jobContext;
    private readonly TimeProvider _time;
    private readonly SagaCorrelationCache _correlationCache;
    private readonly IOptions<WarpConfiguration> _config;

    public SagaHandlerProxy(/* … all deps … */) { /* … */ }

    public async Task HandleAsync(TMessage message, CancellationToken ct)
    {
        var key = _correlationCache.GetCorrelationKey(message);
        var lockName = $"warp:saga:{typeof(TSaga).FullName}:{key}";

        await using var handle = await _semaphoreProvider.TryAcquireAsync(lockName, 1, TimeSpan.Zero, ct);
        if (handle == null)
        {
            _jobContext.Outcome = BuildRequeueOutcome(typeof(TSaga).Name, key, _time.GetUtcNow().UtcDateTime);

            return;
        }

        try
        {
            var saga = await _store.Load<TSaga>(key, ct);
            var isNew = saga == null;
            if (isNew)
            {
                if (!HasStartsSagaAttribute(typeof(TMessage)))
                {
                    _jobContext.Outcome = BuildNoSagaOutcome(typeof(TSaga).Name, key);

                    return;
                }

                saga = new TSaga { CorrelationKey = key };
            }

            await _inner.HandleAsync(saga, message, ct);

            if (isNew)
            {
                _store.Add(saga);
            }
            else
            {
                _store.Update(saga);
            }

            if (saga.IsComplete)
            {
                MarkCompletion(saga);                // CompletedAt, ExpireAt
            }

            try
            {
                await _store.SaveChangesAsync(ct);
            }
            catch (DbUpdateConcurrencyException)
            {
                _jobContext.Outcome = BuildVersionConflictOutcome(typeof(TSaga).Name, key, _time.GetUtcNow().UtcDateTime);
            }
        }
        catch when (!ct.IsCancellationRequested)
        {
            throw;                                   // let the standard pipeline mark the job failed
        }
    }

    /* helpers … */
}
```

No `IServiceProvider` injection (memory: `feedback_no_service_provider_injection`).

### Pipeline behavior interaction

The proxy is downstream of all `IPipelineBehavior<TMessage, Unit>` registrations. That means existing addons compose naturally:

- **Retry** addon — saga handler exceptions trigger retry (good — retry will re-acquire the mutex, reload the saga, replay)
- **Concurrency** addon (`[Mutex]`/`[Semaphore]` on the message) — runs *before* the saga proxy. Independent slot pool.
- **CircuitBreaker** addon — wraps the entire dispatch including the saga proxy

No special wiring needed in the pipeline; the proxy is just the innermost handler.

### Design influences

Studied locally: Wolverine (`~/source/repos/wolverine` — `src/Wolverine/Saga.cs`, `src/Wolverine/Persistence/Sagas/SagaChain.cs`, `src/Persistence/Wolverine.RDBMS/Sagas/SagaTableDefinition.cs`, `src/Persistence/EfCoreTests/Sagas/SagaDbContext.cs`). Cross-referenced from prior knowledge: NServiceBus (`Saga<TData>` + `IAmStartedByMessages<T>` + `ConfigureHowToFindSaga`), Rebus (`Saga<TData>` + `IAmInitiatedBy<T>`), MassTransit (state-machine DSL — explicitly out of scope).

Where Warp diverges:

- **Mutex-based serialization** (`IWarpSemaphoreProvider` lock per `(SagaType, CorrelationKey)`) replaces the optimistic-only model every other library uses. Warp already owns a distributed-lock primitive; using it removes the conflict-and-retry spin entirely. Version-bump optimistic concurrency is retained as defense-in-depth.
- **Separate `ISagaHandler<TSaga, TMessage>` interface** instead of Wolverine's "methods on the Saga subclass" model. Reasons: (a) fits the existing `IMessageHandler<T>` DI shape, (b) keeps `Saga` POCO-clean so JSON serialization stays trivial, (c) constructor-injects dependencies the way the rest of Warp does. Closer to NServiceBus/Rebus's `Saga<TData>` split.
- **Single `Saga` table with `Type` discriminator** instead of Wolverine's one-table-per-saga-type. Mirrors Warp's other addon entities (`ConcurrencyLimit`, `CircuitBreakerState`).
- **`[Correlate]` attribute on the message property** instead of NServiceBus's `ConfigureHowToFindSaga` fluent mapper or Wolverine's name-convention fallback chain. Single mechanism, no magic.
- **String-only correlation keys** for v1. Wolverine supports `Guid`/`int`/`long`/string/strong-typed IDs; that flexibility is out of v1 scope. Users coerce.
- **Runtime reflection + per-type cache** for handler dispatch (matches Warp's existing `JobDispatcher` pattern). Wolverine source-generates this — a v2 toolkit-wide optimization.
- **`NotFoundAsync` override hook** (lifted from Wolverine's `NotFound` method). Default is `Failed` outcome; users override to ignore, compensate, or log-and-skip.

Deferred from Wolverine for v2/v3: `TimeoutMessage` base class with auto-schedule, name-convention correlation, source-gen dispatch, non-string correlation IDs, state-machine DSL (MassTransit-style).

## Out of scope (deferred or explicitly rejected)

- **State-machine DSL** (MassTransit-style `Initially`/`During`/`Finally`). Explicitly deferred to v3 if there's demand — `ISagaHandler<,>` covers the common cases.
- **Saga timeouts** (auto-fire a message if no progress in X). Users can do this manually with `_publisher.Schedule(new TimeoutCheck { CorrelationKey = ... }, ScheduleTime)` and a saga handler for `TimeoutCheck`. v2.
- **Compensating actions on saga failure.** Users can `try/catch` in the handler or emit compensation messages explicitly. The framework doesn't auto-rollback.
- **Dashboard pages for sagas.** v2.
- **Event-sourced sagas** (each message appends an event; state is replay). v3 if ever — the row-based model is the default in every major library.
- **Multi-key correlation** (`[Correlate]` on multiple properties on the same message). v1 single key only.
- **`ISagaHandler<TSaga, IJob>`** — sagas only attach to `IMessage`, not `IJob`. Rationale: jobs are single-handler, single-shot — they don't fit the multi-message correlation shape. Users who want job-driven sagas wrap their job in a message and publish it.
- **Per-`(saga, message)` `[StartsSaga]` granularity** — v1 makes the attribute message-level only. If you need "only the OrderPlaced message can start the saga, not PaymentCaptured", you put `[StartsSaga]` on `OrderPlaced` and not on `PaymentCaptured`. PaymentCaptured arriving before OrderPlaced → job fails with a clear log message. v2 may add method-level granularity.
- **Optimistic concurrency without the mutex.** The mutex is required — version conflicts are defense-in-depth only.
- **Custom correlation strategies** (`ConfigureHowToFindSaga` style). v2.
- **Cross-saga messages** (one message routes to multiple saga types). Each `ISagaHandler<TSagaA, M>` and `ISagaHandler<TSagaB, M>` already produce separate `IMessageHandler<M>` proxy registrations and therefore separate routed child jobs — this works out of the box.

## Risks

- **Mutex starvation.** A handler that throws and retries can hold the slot through retries via the existing concurrency-style "release on handler exit" semantics. Slot is released on handler exit (success or fail), retries re-acquire fresh. This is the same model as `[Semaphore]`'s "cap currently-executing handlers, not living jobs" decision (`docs/specs/2026-05-08-semaphore.md:270`). Same characterization test as Concurrency applies.
- **Saga table grows unbounded if `IsComplete` is never set.** Mitigation: docs warn that `IsComplete = true` is the only signal; sagas without it live forever until manually cleaned. Future v2 dashboard surfaces "oldest open saga age" metric.
- **JSON-state schema drift.** When `TSaga` adds a property, deserializing old `StateJson` produces default values. Acceptable — users manage their own state-shape evolution. Documented.
- **Unique filtered index portability.** Filtered/partial indexes work on both PG (since 9.0) and SQL Server (since 2008). EF Core supports both via `.HasFilter(...)`. Integration tests on both backends prove it.
- **Migration burden.** Same as `ConcurrencyLimit` — users run `dotnet ef migrations add UpgradeWarp` themselves. Documented in the upgrade notes.
- **Reflection cost on `[Correlate]` lookup.** Cached per-type after first access; no impact on hot path beyond the dictionary lookup.
- **Proxy registration order.** `AddSagaHandler` must be called *after* `AddWarp` so that `IMessageHandler<TMessage>` registrations from sagas don't get overwritten. Documented; the registration extension's xml-doc says so explicitly. Future improvement: validate at startup that there's at most one proxy per `(TSaga, TMessage)` pair.
- **Saga handler invoked concurrently with the message's own `IMessageHandler<T>` (if user registered both).** Each registration produces a separate child Kind=Job from MessageRouter. They are unrelated. Documented.

## Verification

- A new saga is created when a `[StartsSaga]` message arrives with an unseen correlation key.
- An existing saga is loaded and mutated when a subsequent message with the same correlation key arrives.
- Two messages for the same saga arriving concurrently are serialized (`MutexHeld_SecondMessageRequeues` — the second one logs `Requeued — saga 'X' busy for 'Y'` and the worker picks it up on the next iteration).
- A non-`[StartsSaga]` message for an unknown correlation key fails the job with `No saga for 'X' with key 'Y'` log by default.
- An `ISagaHandler<,>` that overrides `NotFoundAsync` to set `context.Outcome = { State = Deleted, … }` causes the missing-saga case to log-and-skip instead of failing.
- An `ISagaHandler<,>` that overrides `NotFoundAsync` and leaves `context.Outcome` untouched yields the default `Failed` outcome (override is opt-in only).
- `saga.IsComplete = true` sets `CompletedAt`, `ExpireAt`, and the row is deleted by `ExpirationCleanup` after `SagaExpirationTimeout`.
- Optimistic concurrency: `DbUpdateConcurrencyException` from `SaveChanges` routes to a `Requeued — version conflict` outcome (unit test mocks the conflict; integration not feasible since the mutex prevents it in practice).
- State JSON round-trips correctly for `string`/`int`/`bool`/`DateTime`/`Guid`/nested object properties on a sub-typed `Saga`.
- Two distinct saga types correlating on the same key are independent (`OrderSagaA` and `OrderSagaB` keyed on `"O-1"` produce two rows).
- The unique filtered index allows two completed sagas + one new live saga with the same `(Type, CorrelationKey)` to coexist (so re-using a correlation after a saga completes is legal).
- End-to-end integration test on both backends: 3-message `OrderSaga` flow (`OrderPlaced` → `PaymentCaptured` → `InventoryReserved`) completes with `IsComplete = true`.
- Cleanup integration test on both backends: a completed saga with `ExpireAt < now` is removed by `ExpirationCleanup`.
- Telemetry test: `warp.sagas.started`/`completed`/`requeued` counters increment on the expected paths.
- Full test suite passes on both Postgres and SQL Server CI matrices.

## Implementation batches

1. **`Saga` entity + `ISagaStore` + `AddSagaEntity`.** No dispatch yet. Just the storage primitive end of the system: entity, EF Core configuration with filtered unique index, `ISagaStore` + `SagaStore<TContext>` (Load/Add/Update/SaveChangesAsync), row-lock tag constant. Unit tests for `SagaStore` CRUD on both backends. Verify the filtered unique index works (two completed + one open is legal; two open is rejected).
2. **`Saga` base class, `[Correlate]`, `[StartsSaga]`, `ISagaHandler<,>`, `SagaCorrelationCache`.** Public-API shapes only — no dispatch wiring. Unit tests for the cache: throws on missing `[Correlate]`, finds the property, caches.
3. **`SagaHandlerProxy<,>` + the mutex acquisition + outcome routing.** The runtime guts. Unit tests with a fake `IWarpSemaphoreProvider` and fake `ISagaStore`: covers (a) acquire-and-handle happy path, (b) mutex busy → requeue outcome, (c) no-saga + no-`[StartsSaga]` → failed outcome, (d) version conflict → requeue outcome.
4. **`AddSagas()` + `AddSagaHandler<THandler>()` registration extensions.** The DI wiring. Unit tests verify (a) `AddSagaHandler` registers the proxy as `IMessageHandler<TMessage>`, (b) `AddSagaHandler` registers `ISagaHandler<TSaga, TMessage>` resolvable from DI, (c) `AddSagaHandler` throws when the type doesn't implement `ISagaHandler<,>`, (d) `AddSagas` contributes the entity configurator exactly once.
5. **Row-lock interceptor extension + `ExpirationCleanup` extension.** PG + SQL Server row-lock interceptors handle the saga tag. `ExpirationCleanup` adds a guarded Saga cleanup pass. Unit tests on both providers for the tag rewrite. Integration test: a completed saga past its `ExpireAt` is cleaned.
6. **Integration tests on both backends** via `WarpTestServer`. Full 3-message `OrderSaga` flow. Concurrency test: 5 messages for the same saga published concurrently all complete in serial. New-correlation + re-using a completed correlation. Cleanup. Telemetry counters.
7. **Telemetry + docs.** `warp.sagas.*` counters, `warp.saga` activity, doc page `website/docs/features/sagas.md`, `CLAUDE.md` update, `README.md` update.

# Plan: Sagas — long-lived correlated state for message-driven workflows

Spec: `docs/specs/2026-05-12-sagas.md`. JSON sidecar: `docs/specs/2026-05-12-sagas.json`.

7 batches. Subagent path applies (manifest > 6, security_impact = none).

## Implementation deltas vs. plan (recorded 2026-05-13)

During implementation the design pivoted on several points. The batch descriptions below remain as historical context; the items here are what actually shipped.

- **Delete-on-completion replaces keep-and-expire.** No `IsComplete`, `CompletedAt`, `ExpireAt`, or filtered unique index. `Saga.MarkCompleted()` causes the proxy to `Remove` the row in the same `SaveChanges`. Unique index is plain `(Type, CorrelationKey)`. Consequence: **Batch 5 (`ExpirationCleanup` saga branch) is dropped entirely** — no saga rows survive completion, so nothing to cull. `WarpConfiguration.SagaExpirationTimeout` and the `IOptions<WarpConfiguration>` dependency on `SagaHandlerProxy` are also dropped.
- **Activity log via `SagaJobLink` extension table.** A `(SagaId, JobId, CreatedAt)` row is written by the proxy on every invocation. FK cascade to `SagaState` is the belt-and-braces cleanup backstop; the proxy and `SagaCommandService.ForceComplete` stage `RemoveRange` in the change tracker so deletes ride the saga's `SaveChanges`.
- **Dashboard UI shipped in v1** (originally deferred). Pages: `SagasListPage`, `SagaDetailPage` under `/warp/sagas`. Hide-on-404 nav. Endpoints in `WarpEndpoints.cs`: list, types, stats, by-id, activity, force-complete. Services: `ISagaQueryService` + `SagaQueryService<TContext>`, `ISagaCommandService` + `SagaCommandService<TContext>`.
- **Typed correlation keys (`Saga<TKey>` for `string`/`Guid`/`int`/`long`)** shipped in v1 (originally deferred). `SagaCorrelationKeyConverter` canonicalizes to the single `varchar(200)` column; `Saga<TKey>.Key` is the typed projection.
- **`ITimeoutMessage`** (originally v2/v3) shipped in v1. `Publisher.Publish(timeoutMsg)` auto-schedules at `now + Delay`. Timeout-after-completion: proxy silently sets `Outcome = Deleted` instead of `Failed`.
- **`SagaPiiCheck`** added (not in original spec). Startup-time guard that rejects `[Correlate]` properties named like PII (Email, Phone, SSN, etc.). Opt-out via `[Correlate(IsAnonymized = true)]`.
- **`AddSagaHandler<T>()` idempotency guard** — throws on duplicate registration rather than silently double-processing every saga message.
- **`SagaState` entity** lives in `Warp.Core.Data.Entities` namespace (not under `Warp.Core.Sagas`) because the existing data-entity folder convention is by folder, not by feature.
- **Telemetry** counters `warp.sagas.started`/`completed`/`requeued` shipped per Batch 7; the `StartSagaActivity` activity helper was not needed and is dropped.

Source of truth for the shipped surface area is `CLAUDE.md` §8.9 and `website/docs/features/sagas.md`. The batch descriptions below describe the original plan, not the final code.

Pure addition. No renames, no removals, no compatibility shims. Batches build bottom-up: storage primitive (Batch 1) → public surface (Batch 2) → dispatch runtime (Batch 3) → DI wiring (Batch 4) → cross-cutting plumbing (Batch 5) → integration (Batch 6) → telemetry + docs (Batch 7). Each batch leaves the build green and the existing test suite green.

## Batch 1 — `Saga` entity + `ISagaStore` + `AddSagaStateEntity`

**Tests (new) — `src/tests/Warp.Tests/Features/Sagas/SagaHandlerProxyTests.cs` (NoDb, with fakes):**

Use `FakeSemaphoreProvider` (already exists from Concurrency work, `src/tests/Warp.Tests/Fixtures/FakeSemaphoreProvider.cs`) and a new `FakeSagaStore` for in-memory CRUD.

- `HappyPath_NewSagaCorrelation_StartsSagaMessage_CreatesAndInvokesHandler` — proxy acquires mutex, sees no saga, sees `[StartsSaga]`, calls handler, saves; FakeSagaStore has one Added saga.
- `HappyPath_ExistingSaga_LoadsAndInvokesHandler` — proxy acquires mutex, loads existing, calls handler with the loaded state, saves; store has one Updated.
- `MutexBusy_SetsRequeueOutcome` — semaphore returns null handle; proxy sets `_jobContext.Outcome = { State = Enqueued, ScheduleTime = now, ClearHandlerType = true, LogMessage }`; handler never invoked.
- `NoSagaAndNoStartsSaga_DefaultNotFound_SetsFailedOutcome` — store returns null, message lacks `[StartsSaga]`, handler uses default `NotFoundAsync`; outcome is `State = Failed` with clear log message; `HandleAsync` never invoked.
- `NoSagaAndNoStartsSaga_OverriddenNotFound_HandlerObservesContextAndSetsDeletedOutcome` — handler's `NotFoundAsync` override sets `context.Outcome = { State = Deleted, LogMessage = "silently skipped" }`; assert the proxy honors that outcome and does not overwrite it; `HandleAsync` still never invoked.
- `IsCompleteTrue_SetsCompletedAtAndExpireAt` — handler sets `saga.IsComplete = true`; proxy populates `CompletedAt = now` and `ExpireAt = now + cfg.SagaExpirationTimeout`.
- `VersionConflictOnSave_SetsRequeueOutcome` — FakeSagaStore throws `DbUpdateConcurrencyException` from `SaveChangesAsync`; proxy catches and sets requeue outcome with version-conflict log message.
- `HandlerThrows_PropagatesException` — proxy rethrows so the normal pipeline marks the job failed; mutex is released (finally).
- `MutexReleasedAfterHandlerSuccess` — handle disposed even on success path (assert via `FakeSemaphoreProvider.ActiveHandles == 0` after).

**Files (new, test infra):**

## Risks (cross-batch)

- Adding `Saga` to `TestContext` early (Batch 1) but `AddSagas()` later means the `ISagaStore` won't resolve in earlier batches' integration tests — fine because earlier batches don't run integration scenarios.
- Provider-specific row-lock interceptors may need a different code shape than expected. Re-read both files before Batch 5; adapt the plan if the existing logic uses dispatch tables vs. inline switches.
- `SagaServiceConfiguration` registers `IMessageHandler<TMessage>` for each saga proxy. If the user *also* registers a plain `IMessageHandler<TMessage>` for the same message, `MessageRouter` creates two child Kind=Job rows — independent. Documented but not enforced in v1.

# DB-push perf results

Generated by `Warp.PerfTest`. Two modes:

```bash
# Workload mode (default): processes N jobs in two bursts with an idle gap, measures total queries.
dotnet run --project src/benchmarks/Warp.PerfTest/Warp.PerfTest.csproj -- --jobs 1000

# Idle mode: boots a server with no workload, measures queries/sec emitted by background tasks.
dotnet run --project src/benchmarks/Warp.PerfTest/Warp.PerfTest.csproj -- --mode idle --idle-seconds 30
```

The idle mode uses an `ActivityListener` on Npgsql's activity source, so it counts every command —
including ones issued via raw `DbCommand.CreateCommand()` (Warp's atomic SQL methods, the notification
transport's `pg_notify`) that bypass EF Core's command interceptor. This matters because the legacy
counter undercounted those by ~30%.

## Idle queries/sec (PostgreSQL, 2026-05-14)

How many DB commands an idle Warp server emits per second, by configuration. Measured over a 30s
window after a 3s warm-up. Defaults only — only `UseDispatcher` and `UseDatabasePush()` vary.

| Scenario          | Dispatcher | Push  | Queries/sec | Notes |
|-------------------|------------|-------|------------:|-------|
| `workers-poll`    | off        | off   |     **6.4** | no dispatcher consolidation, polling-only — busiest |
| `workers-push`    | off        | on    |     **3.1** | push reduces poll-cycles; per-worker fetches remain |
| `dispatcher-poll` | on         | off   |     **4.5** | dispatcher cuts UPDATE chatter but polling still fires |
| **`dispatcher-push`** | **on**  | **on**|     **1.2** | the recommended config for any non-trivial deployment |

At idle the floor on the recommended `dispatcher-push` config is dominated by:

- **Heartbeat** every 3s — 1 query/tick (CTE+JOIN that folds the server UPDATE, paused_at read, and
  worker_group pause-state read into one round-trip).
- **xact-scoped advisory lock acquires** for server tasks — `pg_try_advisory_xact_lock(...)` per
  ServerTaskLoop iteration that holds a lock. The release is implicit on COMMIT, so no separate
  unlock query.
- **ScheduledJobActivation** every 5s — atomic `UPDATE ... RETURNING queue` that flips due
  `Scheduled` rows to `Enqueued` and streams back distinct queues for notification fan-out.
- **MessageRouter** every 30s (default bumped by `UseDatabasePush()`) — advisory lock + 1 SELECT
  probe per tick when no messages are present. Lock is necessary for multi-server correctness since
  `Job` has no concurrency token.

For deployments using `IMessage` heavily and running multiple servers, MessageRouter's advisory lock
becomes a throughput ceiling (single-server serialization). The right fix is an atomic
UPDATE-RETURNING claim like `WarpDispatcher.ClaimEnqueuedJobsAsync` plus stale-message recovery —
tracked as a follow-up; not on this branch.

## Workload duration (PostgreSQL, 2026-04-20)

Generated with `--mode workload --jobs 1000`.

## Scenario

Each scenario processes 1000 `EmptyRequest` jobs in two bursts of 500, with a 15s idle gap between them. Config: 5 workers, production-realistic polling (`PollingInterval=1s`, `MaxPollingInterval=30s`, `PollingIntervalFactor=2.0`). The idle gap lets polling backoff grow — so the second burst pays the full wake-up cost when push is off.

- **workers-poll** — `UseDispatcher=false`, no push. Individual workers each poll independently.
- **dispatcher-poll** — `UseDispatcher=true`, no push. One batched fetch per group, exponential backoff on idle.
- **dispatcher-push** — `UseDispatcher=true`, `opt.UseDatabasePush()` in the worker setup lambda. Push wakes the dispatcher immediately on every enqueue, bypassing the idle backoff sleep.

## Results (PostgreSQL, 2026-04-20)

| Scenario          | Jobs  | Duration | SELECT | UPDATE | INSERT | DELETE | Other | Total |
|-------------------|-------|----------|-------:|-------:|-------:|-------:|------:|------:|
| workers-poll      |  1000 |    34.0s |   1185 |   1275 |   2239 |      1 |     0 |  4700 |
| dispatcher-poll   |  1000 |    33.9s |   1310 |   1611 |   1542 |      1 |     0 |  4464 |
| dispatcher-push   |  1000 |    18.1s |   1492 |   1568 |   1534 |      1 |     0 |  4595 |

Comparison to the 2026-04-19 baseline (pre hand-SQL atomic-claim conversion): the hand-SQL change fuses the legacy SELECT-then-UPDATE claim into a single `UPDATE ... RETURNING/OUTPUT`, so `SELECT` drops sharply (workers-poll 2417 → 1185, dispatcher-poll 1706 → 1310). The per-job "Processing" `JobLog` is now always persisted in the same save that the claim returns from, bumping `INSERT` counts correspondingly. Total command volume is within ±5% of baseline; wall-clock duration and the push-vs-poll ratio are unchanged.

## Takeaways

- **Duration**: `dispatcher-push` processes the workload in **roughly half the wall-clock time** (18.1s vs 34.0s). The gap comes from the idle period — without push, polling backoff grew to near `MaxPollingInterval=30s`, so the second burst waited ~15s for the next scheduled poll. Push wakes the dispatcher instantly.
- **SELECT count**: pure-poll modes (`workers-poll`, `dispatcher-poll`) now run the fewest SELECTs because the atomic-claim path replaces SELECT+UPDATE with a single UPDATE ... RETURNING. Push adds some SELECTs for notification-listener bookkeeping, which is why `dispatcher-push` is the highest SELECT scenario here — but its wall-clock win is still massive.
- **Individual-worker mode does not benefit from push** by design — individual workers would have a thundering-herd problem on each notification, which is why `opt.UseDatabasePush()` only wires the dispatcher for worker-fetch push. `workers-poll` is shown as the no-dispatcher baseline, not a push target.

The more idle gaps your workload has, the bigger the push advantage. This two-burst test underrepresents long-idle production workloads.

## Caveats

- The workload-mode counts above use the EF Core command interceptor and exclude short-lived
  connections used by `PostgresNotificationTransport.PublishAsync` and Warp's raw-ADO.NET atomic SQL
  methods. The idle-mode numbers use the Activity listener on Npgsql's source and capture
  everything, so the two sections aren't apples-to-apples on raw counts.
- The `EmptyRequest` handler is a no-op — real handlers that do their own DB work will shift the ratios.
- Tested on PostgreSQL only. SQL Server Service Broker has different overhead characteristics; rerun against SQL Server to get comparable numbers.

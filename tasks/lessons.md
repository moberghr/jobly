# Lessons Learned

> This file captures patterns and mistakes discovered during AI-assisted development.
> It is read at the start of every `/project:moberg-implement` session.
> Commit this file — it is institutional memory for the team.

## 2026-04-09 — Naming conventions & schema support

- EF Core's `.ToTable(nameof(Entity))` via Fluent API sets the table name with `ConfigurationSource.Explicit`, which prevents `EFCore.NamingConventions` from transforming it (conventions have lower priority than explicit config). To let naming conventions work, don't call `.ToTable()` — let EF Core use the CLR type name as the default (convention-sourced).
- When Respawn's `TablesToIgnore` uses hardcoded table names, they break when naming conventions transform those names. Resolve table names from the EF model instead (`model.FindEntityType(typeof(Entity))!.GetTableName()`).
- Using `entity.Metadata.SetSchema(schema)` sets schema without re-pinning the table name — this is the correct way to assign schema while preserving naming convention freedom. Don't use `.ToTable(name, schema)` which sets both explicitly.
- `null`-coalescing (`??`) on config properties silently discards explicit `null` values. When a user sets `Schema = null` to opt out, `config?.Schema ?? "warp"` ignores their intent. Use `config != null ? config.Schema : "warp"` to distinguish "no config" from "config with null".
- The `SqlServerRowLockInterceptor` hardcoded table names for string replacement. Use regex against the `FROM [table] AS [alias]` pattern instead, so it works regardless of naming convention or schema.

## 2026-04-09 — Multi-server integration tests

- `IBatchPublisher.StartNew()` and `ContinueBatchWith()` do NOT auto-save. Always call `batchPublisher.SaveChangesAsync()` after batch operations. The publisher and batch publisher are separate DI scopes — `publisher.SaveChangesAsync()` does not save the batch publisher's changes.
- Batch continuations are nested batches (Kind=Batch with ParentId=originalBatchId), not direct children. When asserting batch structure, query continuation batch children separately.
- Don't assert that "both servers processed some jobs" in multi-server tests. Warp provides no fairness guarantee — competitive fetch-and-lock means one server can win all fetches. Test correctness (no duplicates), not load distribution.
- Always await cleanup of CancellableRequest after `DeleteJob` — call `WaitForJobState(id, State.Deleted)` to ensure the handler exits before the next test runs.

## 2026-04-09 — Dashboard demo mode & screenshots

- Axios custom adapters receive `config.url` with the `baseURL` already resolved (full path like `/warp/api/status`, not just `/status`). Strip the baseURL prefix before pattern-matching in mock adapters.
- Vite `server.proxy: undefined` inside a `server: {}` object does NOT disable the proxy — the `undefined` value is ignored and defaults apply. To conditionally disable proxy, return separate objects: `server: isDemo ? {} : { proxy: { ... } }`.
- When using `reuseExistingServer: true` in Playwright, stale Vite dev servers on the same port from previous runs will be reused even if the config changed. Use a distinct port for the screenshot Vite server to avoid conflicts.
- `State.Scheduled` does not exist in the frontend type system — scheduled jobs have `currentState: State.Enqueued` with a future `scheduleTime`. The `/jobs/scheduled` endpoint returns enqueued jobs filtered by schedule time on the backend.

## 2026-05-06 — Warp.Http source generator

- `MapMethods(IEndpointRouteBuilder, string, IEnumerable<string>, RequestDelegate)` returns `IEndpointConventionBuilder` (no `.Accepts<>` / `.Produces<>`). The `Delegate`-overload `MapMethods(string, IEnumerable<string>, Delegate)` returns `RouteHandlerBuilder` with the full Minimal API metadata API. To get the metadata methods while keeping our own RequestDelegate, wrap as `(HttpContext ctx) => requestDelegate(ctx)` — Minimal API recognises HttpContext as a known parameter and injects it directly.
- Generators authoring `WriteAsync` calls into emitted code: `HttpResponse.WriteAsync(string)` is an extension method in `Microsoft.AspNetCore.Http.HttpResponseWritingExtensions`. Generated code in user assemblies often lacks the right `using`, so emit fully-qualified: `global::Microsoft.AspNetCore.Http.HttpResponseWritingExtensions.WriteAsync(ctx.Response, ...)`.
- Records with parameterless construction syntax (`record EmptyStream;`) do have a synthesized parameterless ctor. If `BindingEmitter` returns `WholeBody` for empty types on a body-less verb, JsonSerializer fails on the empty body. Fix: empty-target types use `ParameterlessCtor` mode with no binding reads — emit `new TRequest()` and dispatch.
- `WarpGeneratedHttpRegistry` is `[ModuleInitializer]`-populated and never cleared. Tests that read `Snapshot()` directly must not also call `Add()`, or counts in other tests silently shift. Assertions like `descriptors.Count == 2` are global-state-coupled by design — mirror this constraint when adding new tests.
- ASP.NET `BadHttpRequestException` thrown inside a `RequestDelegate` is not auto-translated to a 400 by TestHost (some hosts do, some don't). Wrap the generated body in `try/catch (BadHttpRequestException ex) { ctx.Response.StatusCode = ex.StatusCode == 0 ? 400 : ex.StatusCode; ... }` so behavior is deterministic regardless of host exception middleware.
- Sonar S2699 ("at least one assertion") fires even when the test relies on `[TimedFact]` deadline as the implicit pass criterion. Add an explicit elapsed-time assertion (e.g. `sw.Elapsed.ShouldBeLessThan(TimeSpan.FromSeconds(2))`) or a state-flag assertion to keep the analyzer quiet AND give the test real teeth.
- Source-generator project references with `OutputItemType="Analyzer" ReferenceOutputAssembly="false"` block the test project from instantiating generator types via reflection. To drive the generator from tests via `CSharpGeneratorDriver`, omit `ReferenceOutputAssembly="false"` (single ref serves both as analyzer at compile time AND as a referenced assembly for tests).
- When you put `[WarpHttp...]`-tagged types into a project that's referenced by another project that ALSO runs the Warp source generator, both compilations emit `Warp.Core.Handlers.Generated.WarpMediatorServiceExtensions` and you get CS0436 conflicts. Keep handler types in ONE project of the dependency chain.

## 2026-05-07 — OpenTelemetry instrumentation

- **Producer span ordering invariant**: when adding a Producer-kind activity around a publish operation, capture the caller's `Activity.Current?.SpanId` into a local *before* opening the span. `Job.ParentSpanId` must reference the caller (HTTP request, parent handler), not the one-tick producer span. Otherwise the consumer becomes a grandchild via the producer, burying every trace one level deeper. Tested by `ProducerSpanTests.Enqueue_WithCallerActivity_JobParentSpanIdIsCallerNotProducer`.
- **Iterator instrumentation pattern**: C# iterator methods cannot have `yield return` inside a `catch` block. To classify exceptions in an `IAsyncEnumerable<T>` wrapper, manually drive `GetAsyncEnumerator()` + `MoveNextAsync()` in a `while (true)` loop, with each call wrapped in its own `try/catch` that tags the activity and rethrows. Reference: `MediatorDispatcher.InstrumentStream<TRequest, TResponse>` after `feat/telemetry`.
- **Process-global ActivityListener flakes under xUnit parallelism — fixed via AsyncLocal sentinel scoping**. `ActivitySource.AddActivityListener` registers process-wide; without scoping, every concurrent test's harness captures every other test's spans. Filtering by message-id papers over it but breaks down for assertions like `Captured.ShouldBeEmpty()` or generic span names (`warp.server_task Heartbeat` overlaps with concurrent `WarpTestServer` heartbeats). Real fix: each harness sets a unique `object _sentinel` into a static `AsyncLocal<object?>` at construction, and the listener's `Sample`/`ActivityStarted` callbacks only claim Activities created on threads whose async-flow inherits the same sentinel. Concurrent harnesses see disjoint span sets even though they share the same `ActivitySource`.
- **AsyncLocal mutations inside an async method don't flow back to the caller**. Critical .NET semantic that bit the harness above: setting `AsyncLocal<T>.Value` inside `IAsyncLifetime.InitializeAsync` (or any async method) does NOT propagate to xUnit's caller, so by the time the test method runs the sentinel is gone. Construct AsyncLocal-scoping helpers either in the test class constructor (synchronous; captured into ExecutionContext at the right point) or inline in the test method body before the first `await`. The `ActivityListenerHarness` doc-comment calls this out explicitly so future maintainers don't repeat the trap.
- **Capture failed test names in flake loops; "exit 2" alone is not enough**. When running the test suite in a multi-iteration loop to hunt flakes, capture the full per-run output to numbered log files and grep for `^failed ` to extract the test name. Reference pattern at the end of this lesson. The MTP "summary" line at the tail of stdout only reports counts — if you only inspect the summary you can't tell which test flaked across runs, and pre-existing-flake noise is indistinguishable from regressions you introduced. Watch for the `failed (canceled) <Class>.<Method>` variant emitted by `[TimedFact]` deadlines — a naive `sed 's/(.*//'` strips the test name; use `sed -E 's/^failed (\(canceled\) )?//; s/ +\([0-9].*\)$//'` instead. Reference harness:
  ```bash
  mkdir -p /tmp/warp-test-runs && rm -f /tmp/warp-test-runs/*.log /tmp/warp-test-runs/summary.txt
  > /tmp/warp-test-runs/summary.txt
  for i in $(seq 1 20); do
    log="/tmp/warp-test-runs/run-$i.log"
    dotnet test --no-build --project tests/Warp.Tests/Warp.Tests.csproj 2>&1 > "$log"
    failures=$(grep "^failed " "$log" | sed -E 's/^failed (\(canceled\) )?//; s/ +\([0-9].*\)$//' | tr '\n' ';' | sed 's/;$//')
    failed=$(grep -E "^  failed:" "$log" | tail -1 | awk '{print $2}')
    echo "Run $i: failed=$failed FAILURES=[$failures]" | tee -a /tmp/warp-test-runs/summary.txt
  done
  grep -h "^failed " /tmp/warp-test-runs/run-*.log | sed -E 's/^failed (\(canceled\) )?//; s/ +\([0-9].*\)$//' | sort | uniq -c | sort -rn
  ```
  Saved logs persist in `/tmp/warp-test-runs/`, so you can re-run the aggregation after the loop completes — useful when a long loop's output scrolls past your terminal buffer.
- **Worker–addon dependency wall**: `Warp.Worker` reads retry attempt info from `jobContext.Metadata["RetriedTimes"]` rather than referencing `Warp.Core.Retry.IRetryMetadata` directly. Pin the literal key to the property name with a `nameof(IRetryMetadata.RetriedTimes)` guard test in the test project — the test project may depend on `Warp.Core.Retry` even though `Warp.Worker` cannot. The guard catches compile-time renames; serialization-key changes (e.g. `JsonPropertyName` attribute on the metadata type) are not caught and would silently break worker-side tagging.
- **Don't assert receive-and-process share trace id**: in OTel messaging conventions a Client-kind `receive` span runs under whatever ambient trace the worker is in (no broker-side trace context to inherit), while a Consumer-kind `process` span explicitly parents to the published job's `TraceId`. They share start-time order (receive precedes process) but NOT trace id. Asserting the trace-id equality breaks the test.
- **Exception messages on span status: keep them, but truncate**. A reviewer flagged `activity.SetStatus(ActivityStatusCode.Error, ex.Message)` as a §1.3 PII vector. After consideration the right answer is to keep the message — exception messages are diagnostic information that operators need on the span, the same data already flows to ILogger and `JobLog.Message` rows, and §1.3 is specifically about *job-payload* `Message` fields, not handler-thrown exception messages. Apply a length cap (e.g., `Truncate(ex.Message, 256)`) to bound size; rely on the dedicated `error.type` tag for low-cardinality filtering. **Don't substitute `ex.GetType().Name` for the description** — that strips the actual diagnostic content for a speculative privacy benefit that the parallel logging paths don't honor.
- **Source-generated mediator forces eager handler registration in tests**: `services.AddWarpMediator()` builds `GeneratedMediator` whose constructor pre-resolves an `IRequestHandler<,>` for every `IRequest<>` in the assembly. Tests that need only a single handler must instantiate `new Mediator(provider)` (the reflection-based mediator) directly to avoid registering every handler that exists in the test assembly. Both mediators route through the same `MediatorDispatcher` entry points so instrumentation coverage is identical.

## 2026-05-12 — Sagas addon

- **Notification dispatch from inside handler-scope SaveChanges is mandatory, not optional.** `Publisher.SaveChangesAsync` (`Publisher.cs:272`) captures pending notifications, commits, then fires. Any new addon that commits inside a handler's scope (sagas, future addons that materialize child rows mid-handler) MUST mirror this pattern or push notifications get silently dropped: by the time the worker's outbox `CapturePending` runs, the newly-added rows are `EntityState.Unchanged` and emit nothing. Polling fallback masks the bug — only visible with `UseDatabasePush()`. Test by enabling push and asserting a saga-published `IJob` wakes immediately.
- **`DbUpdateConcurrencyException` does NOT detach the failed entity.** The Modified saga (and any Added rows from `IPublisher.Publish` inside the handler) stay tracked after the throw. The worker's outbox `SaveChanges` will re-attempt and re-throw, overwriting any `Outcome` the handler set in the catch. Pattern: `_context.ChangeTracker.Clear()` before rethrowing from the addon's own `SaveChangesAsync`. The exception is propagated; the tracker is empty so the worker's outbox commits nothing. Test fakes that "clear pending on throw" mirror this real behavior — fakes that just throw without clearing will mask the bug.
- **Routed message jobs must not `ClearHandlerType` on requeue.** `MessageRouter` sets `Job.HandlerType` to the chosen `IMessageHandler<T>` implementation. If a pipeline behavior (or saga proxy) sets `Outcome.ClearHandlerType = true` for a routed message job, the next worker fetch sees `HandlerType == null` and falls back to `JobDispatcher.DiscoverJobHandler(messageType, provider)` — which calls `IJobHandler<>.MakeGenericType(messageType)`. For a `T : IMessage` the constraint `T : IJob` fails at runtime with `TypeLoadException`. The `Concurrency` addon's `ClearHandlerType = true` is correct for `IJob` paths only; new addons emitting outcomes for `IMessage` paths must keep the HandlerType. Saga proxies do this explicitly with a comment justifying the value.
- **Cross-DB filtered indexes are not portable.** `HasFilter("\"is_complete\" = false")` works on Postgres (double-quoted identifier) and breaks on SQL Server (which uses brackets). EF Core stores the filter SQL verbatim. Two ways out: per-provider `HasFilter` from `WarpModelCustomizer` (we don't have provider awareness there), or use a different mechanism. For sagas, the choice was to **delete the row on completion** (Wolverine pattern) — a strict unique index on `(Type, CorrelationKey)` works on both DBs and the row never lives in a "completed-but-around" state. Lesson: when an addon needs "uniqueness on live rows only," prefer designs where the row disappears at end-of-life over filtered indexes.
- **Source generator regenerates child-DB-test classes only when the `[GenerateDatabaseTests]` attribute or base class change is observed.** `[GenerateDatabaseTests]` produces concrete `_PostgreSql` / `_SqlServer` subclasses. If the runtime ordering matters (integration vs unit) the source generator does NOT distinguish — both inherit from the base class verbatim, and the per-class `IntegrationTestBase` does the setup. The CLAUDE.md mention of `FixtureKind.Integration` is stale documentation — the actual generator has only a `WithPush` bool. Read the source generator before assuming attribute parameters.

## 2026-05-12 — Sagas review round 2

- **`StateJson` should hold only subclass state, not base-class fields.** The `Saga` base class properties (`Id`, `CorrelationKey`, `IsCompleted`) and the typed `Saga<TKey>.Key` projection all need `[JsonIgnore]`. The authoritative storage is the `SagaState` row's columns — `SagaStore.Add` reads `saga.Id` and `saga.CorrelationKey` into the row separately, and `SagaStore.Load` reassigns them after JSON deserialize. Without `[JsonIgnore]` they round-trip via *both* JSON and columns simultaneously, bloating `state_json` and creating a property-order-dependent deserialization path that breaks subtly when the JSON's CorrelationKey field happens to deserialize *after* the typed `Key` setter writes its own value. The C1 regression test (`Add_GuidKeyedSagaSubclass_StoresOnlySubclassStateInJson_AndRoundTrips`) asserts the JSON contains neither `"Key"` nor `"CorrelationKey"` nor `"Id"` nor `"IsCompleted"` — only `"Counter"` (the user-defined property).
- **Don't optimize away the "last touched" timestamp.** First-pass instinct was to skip the `UpdatedAt = now` write when the serialized state hadn't changed (saves a no-op SQL UPDATE). But `UpdatedAt` is the operational signal operators query to find stuck sagas (`WHERE updated_at < now() - INTERVAL '7 days'`). The optimization repurposes it from "last touched by a saga message" to "last state change" — which makes a saga actively receiving no-op messages look stuck. Write amplification on no-op handlers is the lesser cost; reverted in round 2. **Rule: timestamp columns whose name implies "last touched" must reflect "last touched", not "last meaningful change". Add a separate `LastChangedAt` if you need the other semantic.**
- **Public types with the same simple name in different namespaces are a slow-burn ergonomic tax.** `Warp.Core.Sagas.Saga` (user-facing base) and `Warp.Core.Data.Entities.Saga` (storage row) both exported as `Saga` forced every file that touched both to alias one with `using SagaEntity = ...`. The cost surfaces in every test, every internal store, and ultimately every user that needs to drop into raw entity queries. Rename to `SagaState` at design time, not after a review round. Conventions to match: `CircuitBreakerState`, `ConcurrencyLimit`, `Counter` — descriptive nouns that don't collide with user-facing API surface.
## 2026-05-12 — Job timeout addon

- **`DateTime` does NOT roundtrip through `MetadataConvert` by default**. `MetadataSerializer` writes typed metadata via `JsonSerializer.Serialize` (DateTime → ISO 8601 string). `NativeObjectConverter.Read` surfaces JSON strings as raw `string`. The original `MetadataConvert.To<T>` had no string→DateTime branch, so `meta.DeadlineUtc` came back null after every JSON roundtrip. Any future metadata interface adding a `DateTime?` property risks silently degrading to null without a direct roundtrip test. Pattern: when adding a non-primitive type to an `IJobMetadata` interface, write a NoDb test that serializes → deserializes → recreates the typed view and asserts the value survives. Cheaper alternative: store as `long? Ticks` to avoid the conversion altogether (longs roundtrip cleanly through `NativeObjectConverter`).
- **Per-timeout `IServiceScopeFactory.CreateScope()` writes saturate the PG container pool**. The first cut of `stats:timeout` opened a fresh DbContext scope per timeout fire (mirroring `CircuitBreakerStore`'s insert-on-conflict path) so the counter would commit regardless of Delete-mode outcome / Fail-mode rollback. Under integration test load — Respawn-bounded `MaxPoolSize`, multiple parallel test classes — this caused `Npgsql.PostgresException: 53300: sorry, too many clients already`, surfacing as cascading `0ms` failures in unrelated tests. Counters must be emitted through an existing DbContext (worker scope or handler scope), not a new scope per fire. Lesson: any "best-effort fresh-scope DB write inside the pipeline" needs a connection-pool budget review before merging.
- **Integration tests must assert the spec, not the symptom they happen to pass**. The first Total-scope test asserted `processingLogs.ShouldBe(3)` — which is satisfied identically by both correct-Total and silently-degraded-to-PerAttempt. A compliance reviewer caught it. The fix asserts `elapsed_first_processing_to_failed < 4s`, which is ~2.5s on the correct path and ~6s on the broken path — that's the discriminator the spec actually demands. Pattern: when a test's purpose is "the new behaviour is materially different from the old behaviour", the assertion has to differentiate the two regimes, not just confirm "something happened".
- **`Warp.Core.Timeout` namespace shadows `System.Threading.Timeout`** in any unqualified `Timeout.X` reference within `Warp.Core`. Build-time CS0234. Production fix on `NullNotificationTransport.cs` was `System.Threading.Timeout.InfiniteTimeSpan`. Spot the shadow at design time when picking namespaces under the root package.
- **DI insertion order = pipeline outer→inner**. `JobDispatcher.cs:115-122` wraps behaviors end-to-start, so `behaviors[0]` (first registered) is the outermost call. Implication for chains where one behavior throws and another catches: the catcher must be registered FIRST so it wraps the thrower. Pin this with a regression test that inspects the resolved `IPipelineBehavior<,>` order (see `TimeoutAddonOrderingTests`) — purely-docs contracts get violated silently on test-server refactors.
- **`FakeTimeProvider` in integration tests requires disabling `StaleJobRecovery`.** When you replace `TimeProvider.System` with a `FakeTimeProvider` so a timeout pipeline's CTS fires on `time.Advance(...)` instead of wall-clock, every advance also ages out `LastKeepAlive` from the worker's perspective. With default `InvisibilityTimeout = 1 minute` and a 60s advance, the in-flight job appears stale and `StaleJobRecovery` re-enqueues it concurrently with the running handler — manifests as extra `Processing` log rows and `Failed` counts that exceed `MaxRetries + 1`. Fix in test setup: `cfg.StaleJobRecoveryInterval = null; cfg.InvisibilityTimeout = TimeSpan.FromDays(365);`. Pattern lives in `TimeoutIntegrationTestsBase.StartWithFakeTime`. The lesson generalizes to any future addon test that drives fake time and runs against the real worker — anything keyed off `LastKeepAlive` or `now - InvisibilityTimeout` becomes a hazard.
- **Addon attributes live on the request type today; "should be on handlers" is a planned refactor.** `[Mutex]`, `[Semaphore]`, `[Timeout]`, `[NoRestart]` are read from `typeof(T)` in their respective publish behaviors — i.e. from the `IJob` / `IMessage` request class, not from the handler class. For `IJob` this is fine (request:handler is 1:1). For `IMessage` it means "attribute applies across all handlers of the message" — rarely the intent. The semantic-correct API is `[Timeout]` / `[Mutex]` on the IJobHandler / IMessageHandler class (read at pipeline time via `_jobContext.HandlerType`), which gives per-handler granularity for fan-out events. This is a deliberate cross-addon refactor for later; v1.x addons keep the request-type pattern and bail out on `request is not IJob` for the IMessage handler path. Anyone touching this: align all four addons (Mutex/Semaphore/Timeout/NoRestart) at once and update `ConcurrencyPublishBehavior`/`TimeoutPublishBehavior`/`NoRestartPublishBehavior` to read from `_jobContext.HandlerType` at pipeline time, not from `typeof(T)` at publish time.
- **Worker polling loop now uses `Task.Delay(_, TimeProvider, _)` (was real wall-clock).** Changed in `WarpWorker.ExecuteAsync` as part of this feature so `FakeTimeProvider` can drive both the timeout CTS *and* the worker fetch cadence in lockstep. Production behavior is unchanged — `Task.Delay(TimeSpan, TimeProvider.System, ct)` is equivalent to `Task.Delay(TimeSpan, ct)`. In tests with `FakeTimeProvider`, advancing fake time unblocks the worker's idle wait immediately; the only remaining real-time cost is `Task.Yield()` per poll iteration. Same `TimeProvider` is shared via `WarpSingleWorkerHost`, which passes its injected `TimeProvider` into the `WarpWorker` constructor. The keep-alive monitor's `Task.Delay` (`WarpWorkerService.RunJobMonitor`) is intentionally left on wall-clock because its loop runs concurrently with handler execution and doesn't gate test progress.

## 2026-05-17 — WarpBackgroundService

- **EF entity configurations: static methods on `ServiceConfiguration.cs`, NOT a `Configurations/` folder.** The original plan called for one-config-per-file under `src/core/Warp.Core/Data/Configurations/` mirroring a typical EF Core convention. The codebase doesn't use that pattern — every entity has an `AddXEntity(ModelBuilder, string? schema)` static method on `Warp.Core.ServiceConfiguration` and addons opt in via `WarpConfiguration.EntityConfigurators`. Follow that. Lesson: read existing addon configs (Sagas / Concurrency / RateLimit) before drafting a manifest that invents file layouts.
- **Concrete generic services that need `WarpWorkerConfiguration.ServerId` must live in `Warp.Worker`, not `Warp.Core`.** `Warp.Core` cannot reference `Warp.Worker` (one-way dep). The pattern: define interfaces in `Warp.Core.BackgroundServices`, place generic implementations in `Warp.Worker.BackgroundServices`. Same as `IWarpSqlQueries<TContext>` (interface in Core, impls in providers). Don't try to inject ServerId via a Core-only abstraction; just put the impl in Worker.
- **Row locking via `IWarpSqlQueries.LockXAsync` methods, NOT `TagWith(InterceptorConstants.RowLockTable*)`.** The interceptor-tag pattern called out in CLAUDE.md §1.4 was refactored away — competitive fetches now go through provider-specific SQL methods on `IWarpSqlQueries` (`LockJobByIdWaitAsync`, `LockAllServersAsync`, etc.). When adding row-locked fetches, extend `IWarpSqlQueries` with `LockXByYAsync` methods on both `PostgresWarpSqlQueries` (FOR NO KEY UPDATE) and `SqlServerWarpSqlQueries` (WITH UPDLOCK, ROWLOCK). The compliance reviewer initially pointed at the old TagWith pattern — update CLAUDE.md §1.4 to reflect current state.
- **Heartbeat is Scoped; cross-tick state needs a Singleton bridge.** `Heartbeat<TContext>` is registered Scoped (depends on Scoped `TContext`); every tick creates a new instance. State that must persist across ticks (e.g., `previousHeld` lease set for lease-loss detection) belongs in a Singleton helper (`HeartbeatLeaseTracker`) injected into the Scoped task. Use lock-protected swap-buffers for thread safety. Don't try to register Heartbeat itself as Singleton.
- **`loggerFactory.AddProvider` accumulates permanently across host restarts.** `.NET`'s `ILoggerFactory` has no `RemoveProvider`. Adding a provider per registered background service at host start, then recreating the host in tests, piles up providers indefinitely. Guard with a `HashSet<string>` of category names already registered, or track providers in a list and `Dispose` them on `StopAsync`.
- **Cleanup paths added to existing tasks (ServerCleanup, ExpirationCleanup, WarpServerRegistration) MUST guard with `_context.Model.FindEntityType(typeof(X)) != null`.** New entities are opt-in. Existing deployments that don't register the addon won't have the entity in the model, and `_context.Set<X>()` throws. The architecture reviewer caught this in iteration 2 — `CleanupBackgroundServiceLogsAsync` was running unconditionally. Same guard `ServerCleanup.cs:87` uses for `BackgroundServiceInstance` should apply to every cleanup site that touches opt-in addon entities.
- **State-machine writes in fast-path early-`continue` branches: still emit intermediate status rows.** The lease-loss restart path in `BackgroundServiceSupervisor` originally did `LogLeaseLost → release → continue` without writing `Status=Faulted` or `Status=Restarting`. The dashboard timeline lost the fault event entirely. When refactoring a supervisor loop with fast-path branches, audit each `continue` to ensure all spec-required status transitions still fire — observability trumps loop-flow micro-optimization.
- **Test seam for Singleton cross-Scoped state: `internal SeedForTest(...)`.** When a Singleton holds buffered state that depends on prior Scoped calls (e.g., `HeartbeatLeaseTracker._previousHeld`), `RunHeartbeatOnceAsync` creates a new scope and can't populate the Singleton from outside. Add an `internal SeedForTest(IEnumerable<T> heldNames)` method that pre-populates `_previousHeld`. This eliminated the only §4.5 `Task.Delay(600ms)` regression and turned a flaky time-dependent test into a deterministic one-tick assertion.
- **`FakeTimeProvider` integration in `WarpTestServer` is not yet plumbed.** Tests for time-driven behaviors (5-minute healthy-reset, exponential-backoff curve verification) were deferred because there's no current way to inject a `FakeTimeProvider` into the test server's hosted services and have the supervisor's `_time.GetUtcNow()` observe the advance. The `TimeoutIntegrationTestsBase.StartWithFakeTime` pattern from the timeout feature is the closest analog and would unblock these tests if generalized.
- **Frontend addon discovery uses `/api/addons` (single round-trip), not per-addon `/probe` endpoints.** Commit `9701531` replaced hide-on-404 probes with a single `WarpAddonsInfo` response. When adding a new dashboard surface for an opt-in addon, extend `WarpAddonsInfo` with one `bool` flag populated from DI presence (`serviceProvider.GetService<IFooMarker>() != null`). Frontend reads the flag once at boot and gates the nav entry. Don't add a new probe route.
- **Subagents drift from invented plan terminology to real codebase patterns.** Across 9 batches the subagents repeatedly deviated from plan-specified names/paths in favor of matching what actually exists in the codebase (`BackgroundServiceConfiguration.cs` vs `WarpBackgroundServiceBuilderExtensions.cs`; static methods on `ServiceConfiguration.cs` vs `Configurations/` folder; `IWarpBuilder<TContext>` vs `WarpWorkerConfiguration` for builder extensions). These were all auto-fix-clean deviations but reveal the spec author (me) wasn't grounded enough in actual codebase conventions before drafting. Future spec-authoring: read **at least three** existing addon configurations end-to-end before writing the manifest.
- **Nav-property additions silently break test seeds.** Adding an EF Core nav property (`HasOne(p => p.Server).WithMany().HasForeignKey(p => p.ServerId)`) on a non-nullable FK column turns the relationship into a *required* FK. Tests that previously seeded the dependent entity with random `Guid.NewGuid()` for `ServerId` (and no Server row) keep compiling and may pass under EnsureCreated+model-only configurations, but they fail with FK violations against a real PG/SQL Server schema as soon as the FK constraint lands. The first review subagent reported "all tests pass" against a subset; the full-suite run caught 42 newly-failing tests across 5 test classes. Lesson: when adding a `HasOne`/`HasForeignKey` to existing entities, treat it as a checklist item to audit *every* test class that inserts that entity. Don't trust a "subset of tests pass" claim from a fix subagent — run the full suite before declaring complete.
- **Virtual property + consumer code drift.** `WarpBackgroundService.LogRetentionCountOverride` and `LogRetentionAgeOverride` were declared as virtual properties documented to override the global retention config. `ExpirationCleanup.CleanupBackgroundServiceLogsAsync` read only the global values — the override path was dead from day one and silently. Same pattern as the saga-spec "documented but never implemented" issues. The test-reviewer caught it as a critical finding only because they read both the docs AND the consumer code. Lesson: when adding a virtual property whose documented behavior is "override X", grep the codebase for the consumer that should read it, and assert via test that the override path actually takes effect. Or: write the consumer code first, the property second.

## 2026-04-09 â€” Naming conventions & schema support

- EF Core's `.ToTable(nameof(Entity))` via Fluent API sets the table name with `ConfigurationSource.Explicit`, which prevents `EFCore.NamingConventions` from transforming it (conventions have lower priority than explicit config). To let naming conventions work, don't call `.ToTable()` â€” let EF Core use the CLR type name as the default (convention-sourced).
- When Respawn's `TablesToIgnore` uses hardcoded table names, they break when naming conventions transform those names. Resolve table names from the EF model instead (`model.FindEntityType(typeof(Entity))!.GetTableName()`).
- Using `entity.Metadata.SetSchema(schema)` sets schema without re-pinning the table name â€” this is the correct way to assign schema while preserving naming convention freedom. Don't use `.ToTable(name, schema)` which sets both explicitly.
- `null`-coalescing (`??`) on config properties silently discards explicit `null` values. When a user sets `Schema = null` to opt out, `config?.Schema ?? "warp"` ignores their intent. Use `config != null ? config.Schema : "warp"` to distinguish "no config" from "config with null".
- The `SqlServerRowLockInterceptor` hardcoded table names for string replacement. Use regex against the `FROM [table] AS [alias]` pattern instead, so it works regardless of naming convention or schema.

## 2026-04-09 â€” Multi-server integration tests

- `IBatchPublisher.StartNew()` and `ContinueBatchWith()` do NOT auto-save. Always call `batchPublisher.SaveChangesAsync()` after batch operations. The publisher and batch publisher are separate DI scopes â€” `publisher.SaveChangesAsync()` does not save the batch publisher's changes.
- Batch continuations are nested batches (Kind=Batch with ParentId=originalBatchId), not direct children. When asserting batch structure, query continuation batch children separately.
- Don't assert that "both servers processed some jobs" in multi-server tests. Warp provides no fairness guarantee â€” competitive fetch-and-lock means one server can win all fetches. Test correctness (no duplicates), not load distribution.
- Always await cleanup of CancellableRequest after `DeleteJob` â€” call `WaitForJobState(id, State.Deleted)` to ensure the handler exits before the next test runs.

## 2026-04-09 â€” Dashboard demo mode & screenshots

- Axios custom adapters receive `config.url` with the `baseURL` already resolved (full path like `/warp/api/status`, not just `/status`). Strip the baseURL prefix before pattern-matching in mock adapters.
- Vite `server.proxy: undefined` inside a `server: {}` object does NOT disable the proxy â€” the `undefined` value is ignored and defaults apply. To conditionally disable proxy, return separate objects: `server: isDemo ? {} : { proxy: { ... } }`.
- When using `reuseExistingServer: true` in Playwright, stale Vite dev servers on the same port from previous runs will be reused even if the config changed. Use a distinct port for the screenshot Vite server to avoid conflicts.
- `State.Scheduled` does not exist in the frontend type system â€” scheduled jobs have `currentState: State.Enqueued` with a future `scheduleTime`. The `/jobs/scheduled` endpoint returns enqueued jobs filtered by schedule time on the backend.

## 2026-05-06 â€” Warp.Http source generator

- `MapMethods(IEndpointRouteBuilder, string, IEnumerable<string>, RequestDelegate)` returns `IEndpointConventionBuilder` (no `.Accepts<>` / `.Produces<>`). The `Delegate`-overload `MapMethods(string, IEnumerable<string>, Delegate)` returns `RouteHandlerBuilder` with the full Minimal API metadata API. To get the metadata methods while keeping our own RequestDelegate, wrap as `(HttpContext ctx) => requestDelegate(ctx)` â€” Minimal API recognises HttpContext as a known parameter and injects it directly.
- Generators authoring `WriteAsync` calls into emitted code: `HttpResponse.WriteAsync(string)` is an extension method in `Microsoft.AspNetCore.Http.HttpResponseWritingExtensions`. Generated code in user assemblies often lacks the right `using`, so emit fully-qualified: `global::Microsoft.AspNetCore.Http.HttpResponseWritingExtensions.WriteAsync(ctx.Response, ...)`.
- Records with parameterless construction syntax (`record EmptyStream;`) do have a synthesized parameterless ctor. If `BindingEmitter` returns `WholeBody` for empty types on a body-less verb, JsonSerializer fails on the empty body. Fix: empty-target types use `ParameterlessCtor` mode with no binding reads â€” emit `new TRequest()` and dispatch.
- `WarpGeneratedHttpRegistry` is `[ModuleInitializer]`-populated and never cleared. Tests that read `Snapshot()` directly must not also call `Add()`, or counts in other tests silently shift. Assertions like `descriptors.Count == 2` are global-state-coupled by design â€” mirror this constraint when adding new tests.
- ASP.NET `BadHttpRequestException` thrown inside a `RequestDelegate` is not auto-translated to a 400 by TestHost (some hosts do, some don't). Wrap the generated body in `try/catch (BadHttpRequestException ex) { ctx.Response.StatusCode = ex.StatusCode == 0 ? 400 : ex.StatusCode; ... }` so behavior is deterministic regardless of host exception middleware.
- Sonar S2699 ("at least one assertion") fires even when the test relies on `[TimedFact]` deadline as the implicit pass criterion. Add an explicit elapsed-time assertion (e.g. `sw.Elapsed.ShouldBeLessThan(TimeSpan.FromSeconds(2))`) or a state-flag assertion to keep the analyzer quiet AND give the test real teeth.
- Source-generator project references with `OutputItemType="Analyzer" ReferenceOutputAssembly="false"` block the test project from instantiating generator types via reflection. To drive the generator from tests via `CSharpGeneratorDriver`, omit `ReferenceOutputAssembly="false"` (single ref serves both as analyzer at compile time AND as a referenced assembly for tests).
- When you put `[WarpHttp...]`-tagged types into a project that's referenced by another project that ALSO runs the Warp source generator, both compilations emit `Warp.Core.Handlers.Generated.WarpMediatorServiceExtensions` and you get CS0436 conflicts. Keep handler types in ONE project of the dependency chain.

## 2026-05-07 â€” OpenTelemetry instrumentation

- **Producer span ordering invariant**: when adding a Producer-kind activity around a publish operation, capture the caller's `Activity.Current?.SpanId` into a local *before* opening the span. `Job.ParentSpanId` must reference the caller (HTTP request, parent handler), not the one-tick producer span. Otherwise the consumer becomes a grandchild via the producer, burying every trace one level deeper. Tested by `ProducerSpanTests.Enqueue_WithCallerActivity_JobParentSpanIdIsCallerNotProducer`.
- **Iterator instrumentation pattern**: C# iterator methods cannot have `yield return` inside a `catch` block. To classify exceptions in an `IAsyncEnumerable<T>` wrapper, manually drive `GetAsyncEnumerator()` + `MoveNextAsync()` in a `while (true)` loop, with each call wrapped in its own `try/catch` that tags the activity and rethrows. Reference: `MediatorDispatcher.InstrumentStream<TRequest, TResponse>` after `feat/telemetry`.
- **Process-global ActivityListener flakes under xUnit parallelism â€” fixed via AsyncLocal sentinel scoping**. `ActivitySource.AddActivityListener` registers process-wide; without scoping, every concurrent test's harness captures every other test's spans. Filtering by message-id papers over it but breaks down for assertions like `Captured.ShouldBeEmpty()` or generic span names (`warp.server_task Heartbeat` overlaps with concurrent `WarpTestServer` heartbeats). Real fix: each harness sets a unique `object _sentinel` into a static `AsyncLocal<object?>` at construction, and the listener's `Sample`/`ActivityStarted` callbacks only claim Activities created on threads whose async-flow inherits the same sentinel. Concurrent harnesses see disjoint span sets even though they share the same `ActivitySource`.
- **AsyncLocal mutations inside an async method don't flow back to the caller**. Critical .NET semantic that bit the harness above: setting `AsyncLocal<T>.Value` inside `IAsyncLifetime.InitializeAsync` (or any async method) does NOT propagate to xUnit's caller, so by the time the test method runs the sentinel is gone. Construct AsyncLocal-scoping helpers either in the test class constructor (synchronous; captured into ExecutionContext at the right point) or inline in the test method body before the first `await`. The `ActivityListenerHarness` doc-comment calls this out explicitly so future maintainers don't repeat the trap.
- **Capture failed test names in flake loops; "exit 2" alone is not enough**. When running the test suite in a multi-iteration loop to hunt flakes, capture the full per-run output to numbered log files and grep for `^failed ` to extract the test name. Reference pattern at the end of this lesson. The MTP "summary" line at the tail of stdout only reports counts â€” if you only inspect the summary you can't tell which test flaked across runs, and pre-existing-flake noise is indistinguishable from regressions you introduced. Watch for the `failed (canceled) <Class>.<Method>` variant emitted by `[TimedFact]` deadlines â€” a naive `sed 's/(.*//'` strips the test name; use `sed -E 's/^failed (\(canceled\) )?//; s/ +\([0-9].*\)$//'` instead. Reference harness:
  ```bash
  mkdir -p /tmp/warp-test-runs && rm -f /tmp/warp-test-runs/*.log /tmp/warp-test-runs/summary.txt
  > /tmp/warp-test-runs/summary.txt
  for i in $(seq 1 20); do
    log="/tmp/warp-test-runs/run-$i.log"
    dotnet test --no-build --project tests/Warp.Tests/Warp.Tests.csproj 2>&1 > "$log"
    failures=$(grep "^failed " "$log" | sed -E 's/^failed (\(canceled\) )?//; s/ +\([0-9].*\)$//' | tr '\n' ';' | sed 's/;$//')
    failed=$(grep -E "^  failed:" "$log" | tail -1 | awk '{print $2}')
    echo "Run $i: failed=$failed FAILURES=[$failures]" | tee -a /tmp/warp-test-runs/summary.txt
  done
  grep -h "^failed " /tmp/warp-test-runs/run-*.log | sed -E 's/^failed (\(canceled\) )?//; s/ +\([0-9].*\)$//' | sort | uniq -c | sort -rn
  ```
  Saved logs persist in `/tmp/warp-test-runs/`, so you can re-run the aggregation after the loop completes â€” useful when a long loop's output scrolls past your terminal buffer.
- **Workerâ€“addon dependency wall**: `Warp.Worker` reads retry attempt info from `jobContext.Metadata["RetriedTimes"]` rather than referencing `Warp.Core.Retry.IRetryMetadata` directly. Pin the literal key to the property name with a `nameof(IRetryMetadata.RetriedTimes)` guard test in the test project â€” the test project may depend on `Warp.Core.Retry` even though `Warp.Worker` cannot. The guard catches compile-time renames; serialization-key changes (e.g. `JsonPropertyName` attribute on the metadata type) are not caught and would silently break worker-side tagging.
- **Don't assert receive-and-process share trace id**: in OTel messaging conventions a Client-kind `receive` span runs under whatever ambient trace the worker is in (no broker-side trace context to inherit), while a Consumer-kind `process` span explicitly parents to the published job's `TraceId`. They share start-time order (receive precedes process) but NOT trace id. Asserting the trace-id equality breaks the test.
- **Exception messages on span status: keep them, but truncate**. A reviewer flagged `activity.SetStatus(ActivityStatusCode.Error, ex.Message)` as a Â§1.3 PII vector. After consideration the right answer is to keep the message â€” exception messages are diagnostic information that operators need on the span, the same data already flows to ILogger and `JobLog.Message` rows, and Â§1.3 is specifically about *job-payload* `Message` fields, not handler-thrown exception messages. Apply a length cap (e.g., `Truncate(ex.Message, 256)`) to bound size; rely on the dedicated `error.type` tag for low-cardinality filtering. **Don't substitute `ex.GetType().Name` for the description** â€” that strips the actual diagnostic content for a speculative privacy benefit that the parallel logging paths don't honor.
- **Source-generated mediator forces eager handler registration in tests**: `services.AddWarpMediator()` builds `GeneratedMediator` whose constructor pre-resolves an `IRequestHandler<,>` for every `IRequest<>` in the assembly. Tests that need only a single handler must instantiate `new Mediator(provider)` (the reflection-based mediator) directly to avoid registering every handler that exists in the test assembly. Both mediators route through the same `MediatorDispatcher` entry points so instrumentation coverage is identical.

## 2026-05-12 â€” Job timeout addon

- **`DateTime` does NOT roundtrip through `MetadataConvert` by default**. `MetadataSerializer` writes typed metadata via `JsonSerializer.Serialize` (DateTime â†’ ISO 8601 string). `NativeObjectConverter.Read` surfaces JSON strings as raw `string`. The original `MetadataConvert.To<T>` had no stringâ†’DateTime branch, so `meta.DeadlineUtc` came back null after every JSON roundtrip. Any future metadata interface adding a `DateTime?` property risks silently degrading to null without a direct roundtrip test. Pattern: when adding a non-primitive type to an `IJobMetadata` interface, write a NoDb test that serializes â†’ deserializes â†’ recreates the typed view and asserts the value survives. Cheaper alternative: store as `long? Ticks` to avoid the conversion altogether (longs roundtrip cleanly through `NativeObjectConverter`).
- **Per-timeout `IServiceScopeFactory.CreateScope()` writes saturate the PG container pool**. The first cut of `stats:timeout` opened a fresh DbContext scope per timeout fire (mirroring `CircuitBreakerStore`'s insert-on-conflict path) so the counter would commit regardless of Delete-mode outcome / Fail-mode rollback. Under integration test load â€” Respawn-bounded `MaxPoolSize`, multiple parallel test classes â€” this caused `Npgsql.PostgresException: 53300: sorry, too many clients already`, surfacing as cascading `0ms` failures in unrelated tests. Counters must be emitted through an existing DbContext (worker scope or handler scope), not a new scope per fire. Lesson: any "best-effort fresh-scope DB write inside the pipeline" needs a connection-pool budget review before merging.
- **Integration tests must assert the spec, not the symptom they happen to pass**. The first Total-scope test asserted `processingLogs.ShouldBe(3)` â€” which is satisfied identically by both correct-Total and silently-degraded-to-PerAttempt. A compliance reviewer caught it. The fix asserts `elapsed_first_processing_to_failed < 4s`, which is ~2.5s on the correct path and ~6s on the broken path â€” that's the discriminator the spec actually demands. Pattern: when a test's purpose is "the new behaviour is materially different from the old behaviour", the assertion has to differentiate the two regimes, not just confirm "something happened".
- **`Warp.Core.Timeout` namespace shadows `System.Threading.Timeout`** in any unqualified `Timeout.X` reference within `Warp.Core`. Build-time CS0234. Production fix on `NullNotificationTransport.cs` was `System.Threading.Timeout.InfiniteTimeSpan`. Spot the shadow at design time when picking namespaces under the root package.
- **DI insertion order = pipeline outerâ†’inner**. `JobDispatcher.cs:115-122` wraps behaviors end-to-start, so `behaviors[0]` (first registered) is the outermost call. Implication for chains where one behavior throws and another catches: the catcher must be registered FIRST so it wraps the thrower. Pin this with a regression test that inspects the resolved `IPipelineBehavior<,>` order (see `TimeoutAddonOrderingTests`) â€” purely-docs contracts get violated silently on test-server refactors.
- **`FakeTimeProvider` in integration tests requires disabling `StaleJobRecovery`.** When you replace `TimeProvider.System` with a `FakeTimeProvider` so a timeout pipeline's CTS fires on `time.Advance(...)` instead of wall-clock, every advance also ages out `LastKeepAlive` from the worker's perspective. With default `InvisibilityTimeout = 1 minute` and a 60s advance, the in-flight job appears stale and `StaleJobRecovery` re-enqueues it concurrently with the running handler â€” manifests as extra `Processing` log rows and `Failed` counts that exceed `MaxRetries + 1`. Fix in test setup: `cfg.StaleJobRecoveryInterval = null; cfg.InvisibilityTimeout = TimeSpan.FromDays(365);`. Pattern lives in `TimeoutIntegrationTestsBase.StartWithFakeTime`. The lesson generalizes to any future addon test that drives fake time and runs against the real worker â€” anything keyed off `LastKeepAlive` or `now - InvisibilityTimeout` becomes a hazard.
- **Addon attributes live on the request type today; "should be on handlers" is a planned refactor.** `[Mutex]`, `[Semaphore]`, `[Timeout]`, `[NoRestart]` are read from `typeof(T)` in their respective publish behaviors â€” i.e. from the `IJob` / `IMessage` request class, not from the handler class. For `IJob` this is fine (request:handler is 1:1). For `IMessage` it means "attribute applies across all handlers of the message" â€” rarely the intent. The semantic-correct API is `[Timeout]` / `[Mutex]` on the IJobHandler / IMessageHandler class (read at pipeline time via `_jobContext.HandlerType`), which gives per-handler granularity for fan-out events. This is a deliberate cross-addon refactor for later; v1.x addons keep the request-type pattern and bail out on `request is not IJob` for the IMessage handler path. Anyone touching this: align all four addons (Mutex/Semaphore/Timeout/NoRestart) at once and update `ConcurrencyPublishBehavior`/`TimeoutPublishBehavior`/`NoRestartPublishBehavior` to read from `_jobContext.HandlerType` at pipeline time, not from `typeof(T)` at publish time.
- **Worker polling loop now uses `Task.Delay(_, TimeProvider, _)` (was real wall-clock).** Changed in `WarpWorker.ExecuteAsync` as part of this feature so `FakeTimeProvider` can drive both the timeout CTS *and* the worker fetch cadence in lockstep. Production behavior is unchanged â€” `Task.Delay(TimeSpan, TimeProvider.System, ct)` is equivalent to `Task.Delay(TimeSpan, ct)`. In tests with `FakeTimeProvider`, advancing fake time unblocks the worker's idle wait immediately; the only remaining real-time cost is `Task.Yield()` per poll iteration. Same `TimeProvider` is shared via `WarpSingleWorkerHost`, which passes its injected `TimeProvider` into the `WarpWorker` constructor. The keep-alive monitor's `Task.Delay` (`WarpWorkerService.RunJobMonitor`) is intentionally left on wall-clock because its loop runs concurrently with handler execution and doesn't gate test progress.
